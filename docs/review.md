# Review Of The Existing Plan

## 结论

这份 `/Users/xiaomo/Desktop/舒适UI_SDXL_去水印完整方案_核验版.md` 不是坏方案。

但它解决的是：

- 现有系统里的视频去水印链怎么安全接入

它没有解决：

- 一个独立的、多平台可接入的去水印 API 系统该怎么定义和落地

所以结论不是“直接照着做”，而是：

- 保留它的工程约束思想
- 放弃它作为独立项目总方案的定位

## 这份方案好的地方

1. 边界收得很死，避免把图片和视频一起做爆。
2. 强调回滚、门禁、兼容性，这个方向是对的。
3. 明确重型推理不该直接塞进主服务进程。
4. 对现有链路的风险判断比较现实，没有假装已经可上线。

这些东西都应该继承到新项目里。

## 这份方案不适合你当前目标的地方

### 1. 目标错位

它是“旧系统集成方案”，不是“独立产品方案”。

文档核心词是：

- OpenFang
- shipinbot
- Hand
- bridge
- clean-publish

这意味着它默认依附旧工作流存在。

而你现在要的是：

- 一个独立去水印系统
- 各种平台通过 API 接进来
- 由我们统一处理

这是两件事。

### 2. 业务范围太窄

原文档明确只做视频，不碰图片。

这个选择对旧系统改造是对的，但对独立平台只适合当：

- MVP 实施顺序

不适合当：

- 产品总边界

独立平台必须从第一天就写清楚：

- 视频优先上线
- 图片能力预留接口和状态位

否则后面会把 API 再打碎一次。

### 3. 没有平台级 API 合同

原文档没有回答这些基础问题：

- 外部怎么提交任务
- 同步还是异步
- 怎么查状态
- 怎么拿结果
- 怎么重试
- 怎么传回调地址
- 怎么做鉴权
- 怎么做幂等

没有这些，就不是平台方案。

### 4. 没有租户和配额设计

“各种平台接入”天然意味着：

- 多调用方
- 不同额度
- 不同优先级
- 不同回调地址
- 不同文件保留周期

原文档完全没有这一层。

### 5. 没有统一后端抽象

原文档虽然列了三个后端：

- `cloud_inpaint`
- `comfy_diffueraser`
- `local_fallback`

但它还是站在“当前系统内部已有实现”的角度写的。

独立项目应该直接抽象成：

- provider interface
- routing policy
- capability matrix
- fallback chain

否则后面每加一个新后端都要改主逻辑。

### 6. 没有文件生命周期定义

平台化后必须明确：

- 上传文件存哪
- 临时文件保留多久
- 结果文件保留多久
- 失败任务是否保留中间产物
- 本地磁盘满了怎么办

原文档没有这部分。

### 7. 缺少可运营指标

独立平台至少要定义：

- 成功率
- 平均耗时
- 单任务最大时长
- 后端失败率
- GPU 占用情况
- 队列积压

没有这些，后续只能靠感觉维护。

## 应该怎么改

最简单可行的方向如下：

1. 项目独立，不再依赖现有业务运行时路径。
2. 架构独立，采用 `API + worker + provider adapters`。
3. API 从第一天统一成异步任务模型。
4. MVP 先做视频，但接口层预留图片。
5. 后端路由支持：
   - 强制指定 provider
   - 自动路由
   - 失败降级
6. 文件存储、回调、鉴权、任务状态必须进入一期设计。

## 最终判断

如果问题是：

- “这是不是一个比较好的本地去水印方案？”

答案是：

- 对“旧系统里的本地视频去水印接入”来说，方向基本正确。
- 对“独立去水印平台”来说，不够，还缺平台级需求定义。

所以现在最正确的动作不是继续补这份旧文档，而是：

- 单开仓库
- 重新定义需求
- 把旧文档降级为参考资料
