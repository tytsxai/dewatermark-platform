# 项目启动提示词

你现在位于一个独立项目根目录，项目名是“去水印平台”。

先读这些文件，再开始干活：

1. `README.md`
2. `docs/review.md`
3. `docs/requirements.md`
4. `docs/architecture.md`
5. `docs/api.md`
6. `docs/roadmap.md`

项目目标：

- 做一个独立的本地去水印系统
- 通过 HTTP API 让各种平台接入
- 一期先做视频任务
- 架构采用 `API + worker + provider adapter`
- 不依赖 OpenFang、shipinbot 或任何旧运行时路径
- 最终目标是“用户只上传视频，系统自动完成本地 AI 去水印”
- 不接受“要求用户手工提供水印位置”作为正式方案

硬约束：

- 第一优先级是项目能跑起来
- 保持简单，避免过度设计
- 先实现最小闭环，再接真实 provider
- 不做无关优化
- 不改需求定义，除非发现明显矛盾并先写入文档
- 所有路径都基于当前仓库
- 代码、脚本、配置都要能在本地单机运行

开发顺序必须严格按这个流程推进：

1. 先检查仓库结构和文档是否一致
2. 先实现最小可跑骨架：
   - FastAPI 服务
   - SQLite 数据层
   - job 表
   - API Key 校验
   - `POST /v1/jobs`
   - `GET /v1/jobs/{job_id}`
   - `GET /healthz`
   - worker 轮询
   - fake provider
3. 用 fake provider 跑通完整链路：
   - 创建任务
   - 入队
   - worker 执行
   - 更新状态
   - 产出结果文件
4. 再实现：
   - `GET /v1/providers`
   - `provider=auto`
   - fallback chain
   - `Idempotency-Key`
   - callback retry
5. 再补 provider 探测、doctor、runtime contract
6. 再补本地 AI 主链框架：
   - `.runtime/lock.yaml`
   - `.runtime/models/manifest.yaml`
   - `workflows/`
   - ComfyUI runtime bootstrap
   - ComfyUI `/system_stats` 探活
7. 最后再进入模型安装和真正 AI 执行链接入

实现要求：

- 每做一轮改动，先说明要改什么，再动文件
- 改完后必须运行最小验证
- 优先补能阻塞运行的问题
- 如果发现文档和实现冲突，先更新文档，再实现
- 如果有多个方案，默认选最简单最快能跑的

当前你应该立刻做的事：

- 审查现有目录
- 生成最小 Python 项目骨架
- 生成依赖清单
- 生成 API/worker 启动入口
- 生成 SQLite 初始化逻辑
- 生成 fake provider
- 生成最小自测命令

如果仓库已经完成以上骨架，则下一优先级改为：

- 对齐文档和当前实现
- 完成本地 AI runtime framework
- 在不安装模型的前提下，跑通 ComfyUI runtime / probe / doctor / health
- 只在框架完整后再进入模型阶段

输出要求：

- 先给结果，再简要说明
- 指出关键修改文件
- 没跑通的地方直接说，不要包装
